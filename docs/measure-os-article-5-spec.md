# MEASURE OS 5条 仕様メモ

## 1. 目的

- 属人化された現場業務を再現可能なシステムに置き換える
- 現場の「予告・着手・実績」の事実ログを取得
- 異常・ズレを可視化する

## 2. 管理対象

### 作業単位（work_unit）

本システムにおける1レコードは以下の組み合わせで一意とする：

- 業務ID（task_id）
- 工程ID（process_id）
- リーダー（user_id）
- business_date

この単位で「予告・着手・実績」を1セットとして扱う。  
※すべての記録は会社カレンダーに基づく「営業日（business_date）」単位で管理する。

- 予告（翌日の作業予定）
- 着手（作業開始）責任移転同意必須
- 実績（作業結果）
- 異常検知
- 数値ズレ
- 未押下
- 例外入力（後入力など）

例外判定は、対象のbusiness_dateの業務終了時刻を超えた入力を指す。  
※数値ズレは実績と予告の差分（diff_value）を指し、異常判定は「数値乖離（許容誤差を超えたズレ）」で行う。

### 未入力の定義

以下のいずれかが欠けている場合、未入力と判定する：

- planned_value が存在しない
- actual_value が存在しない
- 着手記録（started_at）が存在しない

※判定タイミング：営業日切替時（day_boundary_time）

### 数値定義

- planned_value：予告数量
- actual_value：実績数量
- diff_value：実績と予告の差分（= actual_value - planned_value）

※diff_value はシステム側で自動計算する。  
diff_value は正負を持つ値とし、プラスは過剰、マイナスは不足を示す。

### 例外入力ルール（重要）

- 後入力・紙運用・通信エラー等は例外扱い
- 必ずメモ入力必須（1行）
- 異常（青）扱いになる

### 青点灯時の選択

- プロセス不備（A）
- 結果不備（B）

※両方選択可能  
最初は「分類のみ」（詳細理由なし）

パターンA（プロセス不備）：

- 入力忘れ
- 手順跳び（後回し）

※詳細理由は OFF（将来 ON）

パターンB（結果不備）：

- 設備・材料トラブル
- 作業ミス・廃棄
- 見立て違い
- 入力・段取りミス
- 突発的な優先変更

※詳細理由は OFF（将来 ON）

### メモ

- 通常異常：パターンAは任意、パターンBは必須
- 例外入力：必須
- メモは短文（10〜30文字程度）で入力

※長文禁止（入力負荷軽減のため）

※内部データ（将来拡張用）

- memo_category：メモ内容を分類するための内部タグ（初期は未使用・自動付与前提）

## 3. 状態管理

※異常発生時刻（anomaly_started_at）は初回検知時のみ記録し、以降は更新しない。  
異常は同一作業に対して重複検知しない（初回のみ記録する）。  
一度異常（blue/red）になったレコードは、事務承認（closed）されるまで normal に戻らない。  
ステータスは後退しない（normal → blue → red → closed のみ）。

### 異常の保持方法

異常は排他的に扱わず、複数同時に保持可能とする：

- 未入力（is_missing）
- 順序不備（is_invalid_flow）
- 数値乖離（is_diff_anomaly）

これらは独立して判定され、同時に成立しうる。  
※表示上の優先順位は UI 側で制御し、データとしては優先順位を持たない。

### 異常の単位

異常は「1作業（予告・着手・実績の1セット）」単位で管理する。

### 自動検知

営業日の切替時刻に自動チェックを行い、前営業日分に以下があれば即青点灯：

- 未入力
- 数値乖離

### ステータス

- 青（猶予）
- 赤（放置）

※内部的には `normal / blue / red / closed` を保持

### 遷移ルール

- 異常検知 → 即青
- 異常発生時に anomaly_started_at 記録

異常とは、未入力、順序不備、または数値乖離が発生した状態を指す。  
異常はシステムが自動判定し、現場の選択（A/B）は分類であり判定には影響しない。

※数値乖離とは：  
`|actual_value - planned_value| > tolerance_value` の状態を指す。

### 赤化条件

営業日の judgement_time（例：13:00）を2回跨ぐと赤。

実装上は以下：

- deadline_at を持つ
- now >= deadline_at → 赤

### deadline_at ルール

次の「営業日の judgement_time」を1回目、さらに次の営業日の同時刻を最終期限とする。  
deadline_at はその2回目の13:00とする。

※営業日計算ルール

- company_calendar に基づく営業日のみをカウントする
- 非営業日はスキップする

例：金曜発生 → 月曜（1回目）→ 火曜（2回目）

### 停止条件

事務の最終承認時のみ消灯（closed）。  
※現場の入力では異常状態は解消されない。

### ステータス更新タイミング

以下のタイミングで状態の再評価を行う：

- レコード作成時
- レコード更新時
- 営業日切替時（day_boundary_time）
- 定時判定時（judgement_time）

## 4. リアルタイム次工程ロック

判定は営業日（business_date）単位で行う。  
※本ロックは入力時のみ適用され、過去データの閲覧は可能とする。

### ロック解除条件

ロックは事務による最終承認（closed）のみで解除される。  
※現場による入力完了では解除されない。  
※異常状態の解消は事務の判断に依存する。

### 現場入力の扱い

現場による不足データの補完は可能とするが、ステータス（blue / red）は変化しない。  
※異常は解消されず、事務承認まで維持される。

### 作業結果の見える化

条件：予告に対して実績数値ズレ  
B選択必須

### 実績入力時

条件：着手未押下  
A選択必須  
責任移転同意必須

### 実績入力時

条件：着手未押下、前営業日実績+次営業日予告なし  
対応：不足データ入力させる  
A選択必須

### 次の営業日の着手時

条件：前営業日実績未完了+次営業日予告なし  
A選択必須  
当日作業開始不可

## 5. 権限設計

### 社労士

- 担当企業の閲覧・操作
- ダッシュボード閲覧
- 機能スイッチ操作

### クライアント企業

#### 経営者

- 閲覧のみ

#### 事務

- 承認
- 修正

#### 班長（現場）

- 入力のみ（工程制限あり）

### 権限移譲

以下で強制移譲：

- 理由選択＋登録時
- 日付跨ぎ

現場は以後編集不可。  
事務に移行。

### 責任移転ログ

責任移譲時は以下を必ず記録する：

- 移転元ユーザー
- 移転先（事務）
- 理由
- 実行時刻

## 6. レポート・監視

### 営業日単位

- 営業日の11:30：青点灯中の作業一覧
- 13時：社労士が赤確認
- 経営者へ報告

### 営業週単位（company_calendar に準拠）

- 青発生件数

### 月次

- 青発生率
- 傾向分析

### データの不変性

現場が後から時刻を書き換えることを防ぐため、`created_at`（サーバー記録時刻）と `started_at / planned_at / actual_at`（現場入力時刻）は別々に保存すること。  
また、各時刻は更新不可とし、修正が必要な場合は新たな記録として扱う。

### 入力ログの保持（監査ログ）

本システムでは、すべての入力操作を監査可能とするため、各レコードに対して以下のログ情報を保持する。

- created_at：サーバーで記録された登録時刻（書き換え不可）
- input_time：実際の入力操作が行われた時刻（クライアント時刻）
- input_type：normal / exception
- input_source：入力経路（web / 紙転記 / 通信再送 など）
- is_exception：例外入力フラグ（後入力・紙運用等）
- anomaly_started_at：異常が初めて検知された時刻（初回のみ記録）

※業務終了時刻を超えた入力は自動的に exception とする。  
これらのログはすべて更新不可とし、修正が必要な場合は新規レコードとして追加することで履歴を保持する。  
created_at は常に input_time 以上の時刻となることを保証する（サーバー受信時刻として扱う）。  
※ is_exception および input_type はシステムが自動判定するものとし、ユーザーによる選択は不可とする。

### 修正履歴（監査用）

事務による修正時は、修正前データを保持したまま新規レコードとして保存すること。  
（before / after の両方を保持）

### 時刻仕様（timezone）

すべての時刻は ISO 8601 形式で保存し、JST（日本時間）を基準として処理すること。  
夜勤や日付跨ぎに対応するため、日付判定は company_settings の day_boundary_time に従う。  
business_date は company_calendar に基づいて算出する。  
business_date はレコード作成時に確定し、その後は変更不可とする。

### business_date の算出関数

business_date は以下のロジックで決定する：

input_time と company_settings.day_boundary_time を用いて算出する。

- input_time < 当日のday_boundary_time → 前営業日
- それ以外 → 当日

※本ロジックは全テーブル共通で使用すること。  
※算出後の business_date は変更不可。

### 業務時間設定（シフト対応）

本システムは、日勤・夜勤・24時間稼働のいずれにも対応するため、営業日の判定および例外判定は以下の業務時間設定に基づいて行う。

- day_boundary_time：営業日の開始時刻（この時刻で business_date が切り替わる）
- work_end_time：当該営業日の業務終了時刻（この時刻を超えた入力は例外扱い）
- judgement_time：異常の赤化判定に使用する基準時刻（例：13:00）

### シフト別の考え方

日勤：

- day_boundary_time = 00:00
- work_end_time = 業務終了時刻（例：17:00）

夜勤：

- day_boundary_time を昼（例：12:00）に設定することで対応する
- 夜勤は「営業日の開始が昼である」として扱う

24時間稼働：

- 明確な締め時刻を必ず設定する（例：08:00）
- 24時間稼働であっても、営業日の区切りは必須とする

### 原則

すべての判定は以下に従う：

- business_date：day_boundary_time に基づく
- 例外判定：work_end_time に基づく
- 赤化判定：judgement_time に基づく

これらはすべて company_settings により企業ごとに設定可能とする。

### 許容誤差（tolerance）

各企業ごとに数値ズレの許容範囲を設定する。

- tolerance_value：許容誤差（例：±2）

※単位は各業務に準拠（個、ケース等）

数値乖離の判定は以下とする：

`|actual_value - planned_value| > tolerance_value` の場合、異常とする。

### 適用単位

tolerance_value は以下単位で設定可能：

- 企業単位（デフォルト）
- 業務単位（task_id）

### 業務終了時刻を超えた入力

`work_end_time`（例：17:00）  
`input_time > business_date + work_end_time`

※判定基準の明確化  
例外入力は以下で判定する：

`input_time > business_date + work_end_time`

※ input_time を基準とし、created_at は使用しない。

### 会社カレンダー

company_calendar により営業日を管理する。

- is_workday = 1：営業日
- is_workday = 0：非営業日（休日）

すべての業務判定（business_date、異常検知、期限計算、レポート）はこのカレンダーに基づいて行う。  
非営業日に入力されたデータは、直前の営業日に紐付ける。

### データ整合性ルール

- actual_value は planned_value が存在する場合のみ入力可能
- 着手（started_at）は1回のみ記録可能
- 各入力は上書き不可とし、修正は新規レコードとして扱う

## 最終まとめ

「事実ログ × 異常検知 × 分類（A/B） × 例外記録」
